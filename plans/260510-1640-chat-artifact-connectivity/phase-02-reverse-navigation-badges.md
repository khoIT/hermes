# Phase 2: Reverse Navigation Badges

## Overview

Persist `sourceThreadId` on segment+campaign POSTs and surface a reverse-link pill on each artifact's detail page. Closes the "no way back to the chat that created me" loop. Boards already store `sourceThreadId` on `PinnedCard` — this phase adds it to segments+campaigns AND lights up all three UI surfaces with a uniform pill component.

## Requirements

**Functional:**
- `CreateSegmentPayload` and `CreateCampaignPayload` accept optional `sourceThreadId`.
- Segments and campaigns DB tables persist a nullable `source_thread_id` column.
- Action cards in chat (`action-card-segment.tsx`, `action-card-campaign.tsx`) pass current `threadId` on confirm.
- Board card overlay shows "💬 from '{title}'" pill when `pinnedCard.sourceThreadId` is set.
- Segment detail header shows "💬 Last asked: '{title}'" pill when `segment.sourceThreadId` is set.
- Campaign detail header shows the same pattern.
- Clicking any pill navigates to `/chat/{sourceThreadId}`.

**Non-functional:**
- Migration is forward-compatible (nullable column with no default backfill needed).
- Pill component is shared across the three surfaces (DRY).
- Thread-title lookup is synchronous from in-memory store (no API call).

## Architecture

```
                    user clicks Confirm in action card
                                |
     action-card-segment.tsx ---+--- action-card-campaign.tsx
                                |           passes { ...payload, sourceThreadId: activeThreadId }
                                v
              segments-client.ts  |  campaigns-client.ts
                  POST body now includes sourceThreadId
                                v
                   catalog-api routes accept the field
                                v
                  drizzle insert -> source_thread_id col
                                v
                  detail GET returns sourceThreadId field
                                v
       segment detail / campaign detail / board card
                  render <SourceThreadPill threadId=… />
                                v
                  click -> navigate(/chat/{threadId})
```

`SourceThreadPill` resolves title via `thread-title-lookup.ts` which reads from `chat-store.ts`'s in-memory thread index. Returns "chat thread" as fallback when title cannot be resolved (orphaned reference safe).

## Implementation Steps

1. **Contracts updates** (15 min)
   - Add `sourceThreadId?: string` to `CreateSegmentPayload`, `Segment`, `CreateCampaignPayload`, `Campaign` types.
   - Re-export through `packages/contracts/src/index.ts` if needed.

2. **DB migration** (20 min)
   - Edit `apps/catalog-api/src/db/schema.ts` to add nullable `source_thread_id` columns on `segments` + `campaigns` tables.
   - Run `pnpm --filter @hermes/catalog-api db:generate` to scaffold migration SQL.
   - Verify migration is `ADD COLUMN ... NULL` only — no constraints, no defaults — so existing rows are unaffected.
   - Apply via `pnpm --filter @hermes/catalog-api db:migrate` against local Postgres.

3. **Catalog-api pass-through** (20 min)
   - In segments controller/service `create` handler, accept the new field on input DTO + insert it.
   - Same for campaigns. Both default to null when caller doesn't supply.

4. **Client pass-through** (10 min)
   - `segments-client.create()` and `campaigns-client.create()` accept `sourceThreadId` and include in POST body.

5. **Action cards pass `threadId`** (15 min)
   - In `action-card-segment.tsx` + `action-card-campaign.tsx`, read the active thread id (props from `<AssistantResponse>`'s parent context — likely available via `messageContext` prop or thread-page state).
   - Confirm handler now: `client.create({ ...payload, sourceThreadId: threadId })`.

6. **Thread title lookup helper** (10 min)
   - Create `thread-title-lookup.ts`. Function: `getThreadTitle(threadId): string | null` — read from `chat-store` in-memory index; return null if not found.
   - Export `useThreadTitle(threadId)` hook variant for React components.

7. **`<SourceThreadPill>` component** (30 min)
   - File: `source-thread-pill.tsx` under `chat-rail/` (new home, since Phase 4 also adds reverse pills).
   - Props: `threadId: string`, optional `variant: 'header' | 'card-overlay'`, `prefix?: string` (default "💬 from").
   - Behavior: if `getThreadTitle(threadId)` returns title, render pill with truncated title (max 40 chars). Click -> navigate `/chat/{threadId}`. Otherwise render fallback "💬 chat thread".
   - Style tokens: `T.n50` background, `T.n200` border, `T.n800` text, 12px font, 4px 10px padding, 10px border-radius.

8. **Mount on detail surfaces** (40 min)
   - **Boards:** in `canvas/detail-page.tsx`, on each PinnedCard render — overlay top-right corner `<SourceThreadPill threadId={card.sourceThreadId} variant="card-overlay" />` when set.
   - **Segments:** in `segments/_components/detail-layout.tsx`, header next to displayName — `<SourceThreadPill threadId={segment.sourceThreadId} variant="header" prefix="💬 Last asked" />`.
   - **Campaigns:** same pattern in `campaigns/monitoring.tsx`. Replaces Phase 1's inline banner with the shared component.

9. **Verification:**
   - Create a new segment from chat -> detail page shows pill.
   - Create a new campaign from chat -> detail page shows pill.
   - Pin a chart from a thread -> board card shows overlay pill.
   - Click each pill -> navigates to source thread.
   - Existing seeded data (no sourceThreadId) renders cleanly without pill.
   - `pnpm typecheck` clean.

## Rollback recipe (red-team mitigation)

If migration partially applies or schema diverges from contracts:

```powershell
# Inspect drizzle migrations table
psql $env:DATABASE_URL -c "SELECT * FROM __drizzle_migrations ORDER BY created_at DESC LIMIT 5;"

# Manual rollback (if needed)
psql $env:DATABASE_URL -c "ALTER TABLE segments DROP COLUMN IF EXISTS source_thread_id;"
psql $env:DATABASE_URL -c "ALTER TABLE campaigns DROP COLUMN IF EXISTS source_thread_id;"

# Mark migration as un-applied
psql $env:DATABASE_URL -c "DELETE FROM __drizzle_migrations WHERE name = '<migration-filename>';"

# Revert contract changes via git
git checkout -- packages/contracts/src/segments.ts packages/contracts/src/campaigns.ts
```

Phase 2's migration is `ADD COLUMN ... NULL` only — fully reversible. No data backfill, no constraint changes.

## Risk Assessment

- **Risk:** Migration runs against shared dev Postgres while another developer is using it.
  - **Mitigation:** ADD COLUMN with NULL is non-blocking; communicate before merging. Use `BEGIN; ALTER TABLE...; COMMIT;` so failure is atomic.
- **Risk:** `getThreadTitle` returns null for fixture threads (in-memory state not hydrated yet).
  - **Mitigation:** Pre-hydrate fixture thread index on app boot (already done by `chat-store`); fallback to "chat thread" if miss.
- **Risk:** action-card components don't currently know the thread id.
  - **Mitigation:** Inspect `<AssistantResponse>` -> `<MessageContext>` plumbing; if no prop chain exists, add a thread-id context provider in `thread-page.tsx`.
- **Risk:** PinnedCard overlay clashes with existing card top-right unpin button.
  - **Mitigation:** Overlay sits in the bottom-left of the card; Phase 5 polishes positioning if conflicts arise.
